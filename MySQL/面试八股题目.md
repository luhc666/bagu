4. 对于查询 `select * from tableA where a = 0 and b < 1 and c = 2 order by d desc` 的索引优化建议：

   **通用最优索引**（大多数 MySQL/InnoDB 场景）：建联合索引 `(a, c, b, d)`，并把 d 做成降序（如果数据库支持降序索引）：`(a, c, b, d desc)`

   **原因**（对应优化器使用规则）：
   - `a = ?`、`c = ?` 是等值条件，放前面能最大化过滤。
   - `b < ?` 是范围条件，放在等值后面；一旦用到范围列，后续列通常就很难再用于继续过滤或天然满足排序。
   - `order by d desc` 想走索引排序，要求 d 在能形成有序扫描的前缀上；但由于有 b 的范围条件，很多情况下无法同时既用索引做范围过滤又完全避免 filesort。把 d 放在 b 后面仍有价值：在某些数据分布/执行计划里能减少回表和排序成本，但不保证永远”无排序”。

   **如果更关心 `order by d desc`**（且 b 的选择性很差）：可考虑 `(a, c, d desc, b)`。这样更利于按 d 直接倒序扫描并尽早取到需要的行（尤其配合 limit 时效果明显）。缺点是 `b < 1` 可能不能在索引层面强过滤，只能回表/ICP 再过滤。

   **最终怎么定：看是否有 LIMIT**
   - 没有 LIMIT：通常优先 `(a, c, b, d)` 走过滤为主，排序交给引擎。
   - 有 LIMIT（比如 top N）：通常优先 `(a, c, d desc, b)` 走”先按 d 找到前 N 个候选，再过滤 b”，常更快。
