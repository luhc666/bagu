# 实习产出总结
## 直播切片项目技术点总结
### 任务触发流程
1. 定时任务：监听开播消息，在开播时开启一个5min的定时任务。
优点：只依赖开播消息和切片开通消息（切片开通消息是为了在直播过程中开启切片任务）
缺点：
	1. 任务的触发源是一个定时任务，需要维护这个定时任务。如果服务重启，需要扫表来重新启动正在进行中的任务。并且服务重启期间的直播片段会丢失。且重启这些未完成任务时，还需要加锁来防止在不同节点上的重复执行。
	2. 开启定时任务之后，无法动态修改定时任务的执行间隔。
	3. 需要用一个分布式锁来防止重复执行

2. 延时消息
使用RocketMQ的延时消息作为触发源，可以将任务的故障恢复转嫁给MQ，MQ的可靠性和稳定性都是可信的。
优点：只依赖开播消息和切片开通消息。不需要自己维护任务状态，MQ天然具备灾备转移能力。
缺点：强依赖MQ延时消息。

3. 如果使用内部的定时任务调度平台，会产生大量的临时任务，并不推荐这种做法。

### 任务解耦
1. 由于切片生成任务的流程很长（共计6次大模型调用），使用RocketMQ将其解耦成两个小任务。实现更轻量化的重试。
2. 如果不解耦，即使将前面的模型的中间结果缓存，重试消息仍然需要与新的消息抢令牌，导致模型资源的浪费。

### 轮转式AB实验
1. 预期通过传统AB实验的方式，找出若干个同质化严重的商家（每组50个），在这些商家上开展AB实验。但是，即使加上很多限定条件（例如相同行业、相同等级），这些商家的日GMV还是会受爆品、是否开播影响，导致实验组和对照组的日GMV的差值能够达到最高50%。
2. 上述问题是由于样本量太小，GMV受随机因素影响大导致的。可以通过在同一直播间中两个模型依次挂车来解决这个问题。在此基础上，由于本实验是为了验证新模型挂车准确率是否能带来GMV的提升，对于那些两个模型匹配结果一致的case，将不纳入实验统计范围。此外，为了抵消随机性带来的影响，实验需要开启一段时间，使用两版模型累计匹配多个切片，在较大量的数据上得到一个较为置信的结论。


## EVQ
### MQ选型
1. 主要从kafka和rocketMQ这两个里面选。kafka的主要优点是高吞吐量，但是缺点是可能会有消息丢失，且无重试机制，需要自己实现。考虑到我们业务的并发量不高（QPS=60），并且调用模型可能失败需要重试，基于以上考虑选择了rocketMQ。
2. 使用异步而不是同步调用劣质模型的原因：商业化的劣质模型的推理时间大概在2-3分钟，如果采用rpc接口调用的话会快速耗尽TomCat线程池导致服务不可用。此外，由于RPC是基于TCP连接的，可能导致TCP连接池被打满，无法再与其他服务建立TCP连接。
### 兜底消息
1. 由于有大概20%的视频会被劣质模型过滤掉，因此采用劣质和优质模型串行化的架构。由于优质模型的触发源强依赖商业化发送回来的消息，如果商业化的服务挂了，会导致我们的服务无法正常运行，所以需要一套兜底机制。
2. 考虑两种方案。一是在创建劣质评测任务之前，把任务持久化到DB，再通过一个定时扫表任务，定期清理那些商业化回调超时的任务。这样有缺点：1.扫表任务太重，而且是分库分表。2.通过扫表得到若干个超时任务后，由于我们的服务是分布式部署，还需要确保一个任务只在一个实例上执行，成本太高。
3. 采用一种轻量化的延时消息方式，在创建劣质评测任务时，同时发送一条mock结果为通过的延时消息。延时时间根据商业化的返回时间而定。（2-3分钟，延时时间定为半小时。）同时需要在消费消息时做幂等。

### 重试机制
1. 调用模型时需要严格限制并发量。由于目前是在调用链入口处做限流，如果原地重试的话，会导致对下游服务造成额外的压力，严重时可能雪崩。如果在每个模型调用前做限流，重试流量可能和新来的流量争抢令牌。

### 数据标注tips（from算法方案）
为了兼顾“可解释”与“可训练”，建议把优质标签拆解成有明确语义边界的内容。否则模型无法理解一个抽象标签是如何定义的。这里每个子标签固定6个字段：
	○ 定义（Definition）：一句话说明“这个标签是什么”。
	○ 命中条件（Must-have）：满足哪些必要条件才算命中（避免“感觉像”）。
	○ 不包含（Must-not）：哪些情况不算（和相邻标签的边界）。
	○ 易混淆点（Confusions）：常见混淆标签&区分关键。
	○ 证据来源（Evidence）：需要哪些证据（帧/ASR/音频特征/时序）。
	○ 边界样例（Edgecases）：最容易争议的例子怎么判。
数据标注标准的质量直接影响模型输出的质量，不好的标准会导致模型输出不稳定，甚至相同的数据在0.1温度下两次推理结果不同，此外，还会出现模棱两可的情况(score_reason解释为高分或者低分都合理)。

# `createTask` 幂等策略总结

## 目标

避免同一个 `photoId` 在 EVQ 劣质链路中被重复调用远端 `createTask`，并尽量保证在远端调用成功但本地状态更新失败时，后续流程仍可继续推进。

---

## 核心思路

采用“**Redis 锁 + DB 状态机**”的组合方式实现幂等控制：

- Redis 锁用于拦截大多数并发触发
- DB 状态机用于承接任务创建过程中的中间状态，避免远端调用成功后本地无记录
- 下游状态流转允许从 `CREATING` 直接推进，用于兜底 `CRC_PENDING` 更新失败场景

---

## 状态定义

### 1. `CREATING`
表示当前记录已抢占创建资格，正在调用远端 `createTask`。

### 2. `CRC_PENDING`
表示远端 `createTask` 已成功，本地已记录，等待 CRC 回调。

### 3. `CREATE_FAIL`
表示远端 `createTask` 明确失败，可作为后续重试入口。

### 4. 终态
- `CRC_FINISH`
- `VALID`

---

## 执行流程

### 1. 入口幂等控制
在 `EvqDelayConsumer` 中先按 `photoId` 加 Redis 锁，避免同一视频并发进入创建流程。

### 2. 创建前先落库
在调用远端 `createTask` 前，先在 DB 中插入一条状态为 `CREATING` 的记录。

这样做的目的是：

- 先占住创建资格
- 即使后续远端调用成功但本地状态更新失败，DB 中仍然保留一条可追踪记录
- 避免“远端已有任务，但本地完全无痕迹”的情况

### 3. 调用远端 `createTask`
执行 CRC 创建任务接口。

### 4. 根据调用结果更新状态
- `createTask` 成功：将状态从 `CREATING` 更新为 `CRC_PENDING`
- `createTask` 明确失败：将状态从 `CREATING` 更新为 `CREATE_FAIL`

---

## 兜底策略

### 1. `CRC_PENDING` 更新失败兜底
如果远端 `createTask` 已成功，但本地将状态从 `CREATING` 更新为 `CRC_PENDING` 失败：

- 不重新调用 `createTask`
- 保留该记录为 `CREATING`
- 后续 CRC 回调允许直接将状态从 `CREATING` 推进到下一状态

即允许以下状态流转：

- `CREATING -> CRC_FINISH`
- `CREATING -> VALID`
- `CRC_PENDING -> CRC_FINISH`
- `CRC_PENDING -> VALID`

这样即使“创建成功后的第二次 DB 更新失败”，下游仍可正常收敛到终态。

### 2. `CREATE_FAIL` 更新失败处理
如果远端明确失败，但更新 `CREATE_FAIL` 失败，则记录仍可能停留在 `CREATING`。  
该场景需要通过日志、告警和后续补偿处理识别，避免记录长期停留在中间态。

---

## 事务策略

`createTask` 相关流程**不使用大事务**包裹“写 `CREATING` + 调远端 + 更新状态”全过程。

原因：

- 本地事务无法回滚远端 `createTask` 的副作用
- 如果远端已成功，而事务最终回滚，会导致“远端已有任务，本地无记录”
- 因此更适合采用“先写中间态，再调远端，再尽力推进状态”的方式处理

---

## 下游状态机要求

下游更新逻辑不能只依赖 `CRC_PENDING`，必须兼容 `CREATING`，否则会出现以下问题：

- 远端已创建成功
- 本地 `CRC_PENDING` 更新失败
- 回调到来时因状态不匹配而无法继续推进

因此所有依赖 CRC 创建结果的后续状态流转，都应允许从 `CREATING` 开始推进。

---

## 当前策略能解决的问题

- 避免大多数并发场景下重复调用远端 `createTask`
- 避免远端已创建成功但本地完全无记录
- 为“创建成功但状态更新失败”提供兜底路径
- 让下游回调能够继续推进状态到终态

---

## 当前策略的边界

- Redis 锁只能减少并发，不能单独作为最终幂等保证
- 如果远端超时且不支持查询，则无法准确判断任务是否已创建成功
- 如果远端不支持幂等键，则超时重试仍可能产生重复远端任务
- 因此本地状态机只能尽量降低风险，无法在远端无幂等能力时做到绝对一致

---

## 最终结论

`createTask` 的幂等策略采用如下方案：

1. 入口按 `photoId` 加 Redis 锁
2. 调用远端前先写 DB，状态为 `CREATING`
3. 远端成功后更新为 `CRC_PENDING`
4. 远端明确失败后更新为 `CREATE_FAIL`
5. 下游状态机允许从 `CREATING` 直接推进到终态，用于兜底 `CRC_PENDING` 更新失败场景

该方案的核心不是追求本地事务上的强一致，而是通过“中间态落库 + 下游兼容推进”来保证整个链路最终可收敛。



# CRC 回调链路幂等方案总结

## 目标

保证同一个 CRC 回调消息在重复投递、并发消费、以及部分状态更新失败的情况下，不会导致 EVQ 链路重复执行或状态错乱，并且能够最终收敛到正确结果。

---

## 核心思路

CRC 回调链路采用“**Redis 锁 + DB 状态机推进**”的方式实现幂等控制：

- Redis 锁用于保证同一个 `photoId` 的回调串行执行
- DB 状态机用于承接 CRC 回调执行过程中的阶段状态
- 下游状态推进允许从中间态继续流转，兜底部分 DB 状态更新失败场景

---

## 锁粒度

在 `EvqCrcEvaluateConsumer` 中，按 `targetId/photoId` 加 Redis 锁：

- 锁 key：`evq:goodQuality:lock:{photoId}`

作用：

- 防止同一个视频的重复 CRC 回调并发执行
- 保证同一时刻只有一个线程推进该视频的 EVQ 状态

前提假设：

- Redis 锁超时时间 >= 下游高耗时调用最大耗时 + buffer

---

## 状态定义

### 1. `CREATING`
已完成 `createTask` 前置占位，或已进入创建流程，但本地状态尚未完全推进。

### 2. `CRC_PENDING`
远端 CRC 任务已成功创建，等待 CRC 回调结果。

### 3. `CREATE_FAIL`
远端 CRC 创建明确失败，可作为重试入口。

### 4. `CRC_FINISH`
CRC 劣质阶段已完成，且未命中 bad tag，可继续执行优质模型评测。

### 5. `VALID`
整条 EVQ 链路已完成，属于最终完成态。

---

## CRC 回调处理流程

### 1. 消费回调消息
`EvqCrcEvaluateConsumer` 收到 CRC 回调消息后：

- 解析 `CrcCommonResult`
- 根据 `photoId` 获取 Redis 锁
- 在锁内执行 `evqCrcCallbackSinglePhoto`

### 2. 终态幂等判断
进入 `EvqCrcCallbackService` 后，先查询该视频是否已经完成整条 EVQ 链路。

幂等判断规则：

- 只将 `VALID` 视为“已完成”
- 如果已经是 `VALID`，则直接跳过，不再重复执行

注意：

- `CRC_FINISH` 不能视为整条链路完成
- 因为 `CRC_FINISH` 仅表示 CRC 已完成，不代表优质模型已成功执行

---

## 状态推进规则

### 场景 1：CRC 命中 bad tag
如果 CRC 回调命中了目标 bad tag，则直接更新为终态：

- `CREATING -> VALID`
- `CRC_PENDING -> VALID`

说明：

- 命中 bad tag 后无需继续执行优质模型
- 此时可直接落最终结果

### 场景 2：CRC 未命中 bad tag
如果 CRC 回调未命中 bad tag，则先更新 CRC 阶段完成状态：

- `CREATING -> CRC_FINISH`
- `CRC_PENDING -> CRC_FINISH`

然后继续执行优质模型评测。

### 场景 3：优质模型评测成功
调用 `evqEvaluateService.evqEvaluateSingleVideo` 成功后：

- `CRC_FINISH -> VALID`

说明：

- 只有优质模型完成后，整条 EVQ 链路才算真正完成

### 场景 4：优质模型评测失败
如果调用 `evqEvaluateService.evqEvaluateSingleVideo` 失败：

- 保持 DB 状态为 `CRC_FINISH`
- 不回滚到 `CREATING` 或 `CRC_PENDING`
- 不改为 `INVALID`

原因：

- CRC 阶段已经真实完成，不应抹掉该检查点
- `CRC_FINISH` 表示“CRC 已完成，但优质模型尚未完成”
- 后续可通过补偿、重试、重复回调等方式继续推进

---

## 兜底设计

### 1. `CRC_PENDING` 更新失败兜底
如果 `createTask` 成功，但本地将状态从 `CREATING` 更新为 `CRC_PENDING` 失败，则回调到来时仍允许：

- `CREATING -> CRC_FINISH`
- `CREATING -> VALID`

这样可以避免“创建成功但状态没更新，导致回调无法继续推进”的问题。

### 2. 优质模型失败兜底
如果在 `CRC_FINISH` 之后调用优质模型失败：

- 保留 `CRC_FINISH`
- 通过日志、监控、补偿任务或后续重试继续推进到 `VALID`

### 3. DB 更新返回 0 行兜底
所有状态更新操作都必须检查影响行数。

如果更新返回 0 行，说明：

- 当前状态不符合预期
- 或记录不存在
- 或已被其他线程推进

此时必须：

- 打 error 日志
- 记录关键上下文：`photoId`、原状态、目标状态、回调内容
- 接入监控/告警，避免状态机静默失效

---

## 幂等判定原则

CRC 回调链路中，幂等的核心不是“禁止所有重复执行”，而是“允许重复回调进入，但状态推进只能发生一次，并最终收敛”。

因此：

- Redis 锁负责串行化
- DB 状态负责判定是否还能推进
- `VALID` 是唯一完成态
- `CRC_FINISH` 是阶段完成态，不是最终完成态

---

## 是否依赖 DB 唯一键

CRC 回调链路主要以“更新已有记录”为主，因此：

- DB 唯一键不是此处幂等的核心保障
- 真正决定幂等性的，是状态判断和条件更新 SQL

唯一键的作用更多在上游 `createTask` 阶段，防止重复插入占位记录。

---

## 是否需要事务

CRC 回调链路**不建议使用大事务**包裹“更新 `CRC_FINISH` + 调用优质模型 + 更新 `VALID`”全过程。

原因：

1. `evqEvaluateService.evqEvaluateSingleVideo` 是高耗时调用
2. 长事务会增加数据库连接和锁持有时间
3. 即使优质模型失败，也不应抹掉 CRC 已完成这一事实
4. `CRC_FINISH` 作为阶段检查点保留下来，更利于补偿和重试

因此更适合采用：

- 先更新到 `CRC_FINISH`
- 调用优质模型
- 成功后再更新到 `VALID`
- 失败则保留 `CRC_FINISH`

---

## 最终状态机语义

- `CREATING`：创建流程中或创建成功但状态未完全推进
- `CRC_PENDING`：CRC 任务已创建，等待回调
- `CREATE_FAIL`：创建失败
- `CRC_FINISH`：CRC 已完成，优质模型可继续执行
- `VALID`：整条 EVQ 链路完成

---

## 最终结论

CRC 回调链路的幂等方案如下：

1. 按 `photoId` 加 Redis 锁，保证回调串行执行
2. 进入回调逻辑后，仅将 `VALID` 视为已完成态
3. 回调状态推进允许：
   - `CREATING -> VALID`
   - `CREATING -> CRC_FINISH`
   - `CRC_PENDING -> VALID`
   - `CRC_PENDING -> CRC_FINISH`
4. 优质模型成功后：
   - `CRC_FINISH -> VALID`
5. 优质模型失败时：
   - 保持 `CRC_FINISH`
   - 不回滚阶段状态
6. 所有状态更新都必须检查影响行数，并对 0 行更新进行日志和告警

该方案的核心是通过“锁串行化 + 状态机条件推进 + 中间态保留”保证 CRC 回调链路在重复回调和部分失败场景下仍然能够最终收敛。
