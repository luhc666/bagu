# 面试八股题目
1. 自增ID不适合分布式数据库，分表分库场景。适合小型项目

2. UUID会影响索引效率，因为UUID是无序的，用一堆无序的ID来构建一个有序的索引目录，性能上肯定有问题

3. JWT存在的问题以及解决方案：

   3.1 token被解密：加盐值（密钥），每个项目的盐值不能一样

   3.2 token被拿到第三方使用：没啥好方法，使用限流

4. SpringBoot自带的定时任务@EnableScheduling存在两个问题。1只适合单体应用不适合集群（但是这个可以通过增加分布式锁来解决），2没法实时更改定时任务和状态。

5. Mybatis一级缓存：在同一个事务中，多次查同一个SQL（即SQL语句完全相同），SQL语句只会执行一次，之后将会直接取缓存中的结果。

6. synchronized存在两个问题：1会导致售卖效率不高。2在多节点的情况下，还是会出现超卖.

7. 缓存穿透的解决方案：

   1. 缓存空对象，优点是实现方便，缺点是额外的内存消耗，以及短期的不一致（比如在redis中存null值后，真的值被保存到数据库了）。
   2. 布隆过滤器，优点是内存占用少，没有多余的key，缺点是实现复杂，且存在误判可能
   3. 增加id复杂度，并做好id格式的校验
   4. 加强用户权限的校验

8. 缓存雪崩的解决方案：
   1. 给不同的key的TTL添加随机值
   2. 利用redis集群提高服务的可用性（主要防止redis服务器宕机的情况）
   3. 给缓存业务添加降级限流策略
   4. 给业务添加多级缓存

9. 缓存击穿的解决方案：
   1. 使用互斥锁，在多个线程同时请求数据时，只能有一个线程进行缓存重建，其他线程等待这个线程缓存重建完成后再查询缓存。

10. 订单表使用自增id存在两个问题：1id的规律性太明显，2受单表数据量的限制（单张表无法保存大量数据，在分库分表的情况下自增id可能会重复）

11. 开启事务，之后再加锁，当锁住的代码执行完成后，再提交事务，因此锁住的代码块执行是在事务之内执行的，可以推断在代码块执行完时，事务还未提交，锁已经被释放，此时其他线程拿到锁之后进行锁住的代码块，读取的库存数据不是最新的。解决方法是在事务开始前加锁。

12. 使用BlockingQueue作为阻塞队列存在的问题：BQ存在于堆中，在高并发场景下，如果不设置BQ的长度上限，可能会爆内存。如果JVM宕机，内存中的数据不会被保存，造成数据不一致的问题。

13. 基于PubSub的消息队列的优缺点：优 采用发布订阅模式，支持多生产、多消费。缺 不支持数据持久化（消息发出如果没有被接受就直接丢弃了） 消息堆积有上限，超出时数据丢失（比如消费者的处理速度慢）
